## Failures

No reward failures or diagnostic-only failures are present in this checked-in sample.

If a future run has failures, reviewers should read them in this order:

1. Reward failures: selected primary rewards and selected diagnostics.
2. Hard policy failures: always reward-relevant.
3. Diagnostic-only failures: useful for understanding behavior drift, especially tool path differences.
4. Raw score failures: the underlying scorer observations.

## Artifact Layout

Generated realtime runs write one self-contained run-level artifact tree:

```text
reports/
  evals/realtime/<run_id>/
    results.json
    results.md
    simulations/
    artifacts/
      <case_id>/<attempt_id>/
        report.json
        report.md
        sim_status.json
        scoring.json
        trace.json
        transcript.json
        tool_calls.json
        audit.json
        final_state.json
        audio/
          manifest.json
          clean_input.pcm
          clean_input.wav
```

The checked-in JSON sample uses the same field names for run metadata, case summaries, reward basis, primary rewards, diagnostics, raw scores, and artifact references. The referenced paths are illustrative and are not expected to exist in the repository.
Attempt directories use short run-local IDs such as `attempt_001`; `sim_status.json` keeps the full realtime runner ID as `realtime_run_id` when deeper trace correlation is needed.

## Caveats

- This sample is source documentation that mirrors generated `reports/evals/realtime/<run_id>/` output.
- It avoids requiring reviewers to spend realtime API credits locally.
- It does not score assistant audio quality, stereo `both.wav`, overlap/interruption metrics, barge-in timing, or full multi-turn Run simulations.
- Realtime transcripts remain diagnostic evidence only. Operational writes still require structured tool results, server-created confirmations, policy revalidation, state-version validation, and audit events.
